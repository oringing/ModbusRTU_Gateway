## 一、P0 级 紧急阻断问题（立即处理）
### P0-001 UART 错误恢复逻辑存在死锁隐患
🔥 **符合分支主题**（系统防御加固 - 故障恢复机制）
涉及模块：Bsp/Src/uart.c 内部 BSP_UART_RecoveryIfNeeded 函数  
问题描述：串口硬件错误后进入自动恢复流程，无重试次数、无超时限制。  
核心风险：串口硬件永久故障时，会陷入无限循环重试，阻塞当前任务正常调度。  
改进建议：  
1. 增加恢复最大重试次数阈值；  
2. 增加单次恢复操作超时判定；  
3. 超限后标记硬件故障状态，停止自动重试，进入系统降级运行逻辑。  
当前状态：**⚠️ 部分验证（2026-05-09）**
- **实现逻辑**：
  - 新增 `s_recovery_retry_count` 计数器，设定阈值 `UART_MAX_RECOVERY_RETRY = 10`
  - 超限后标记 `s_uart_hw_fault = true`，停止自动恢复
  - 故障反馈：LED 快速闪烁 5 次（50ms 间隔），记录 `ERROR_SYSTEM` 日志
  - 自愈逻辑：故障标记后连续接收 5 帧则清除故障标记，恢复正常恢复逻辑
- **测试状态**：⚠️ 部分验证
  - ✅ 手动物理断线场景下 UART 硬件未触发错误标志位，恢复逻辑静默
  - ❌ 代码注入测试未验证熔断计数器累加逻辑
  - ⚠️ 真实硬件故障场景需在工业干扰环境下进一步验证（实验室环境无法可靠触发 ORE/FE/NE）
- **根因分析**：STM32 HAL 库特性，实验室环境无法模拟工业现场电磁干扰导致的硬件错误标志位

### P1-007 寄存器内部写（InternalWrite）与舵机/心跳寄存器的潜在风险
🔥 **符合分支主题**（系统防御加固 - 寄存器与外设交互安全）
涉及模块：`App/Protocol/Src/modbus.c`、`App/Driver/Src/driver_servo.c`、`App/Task/Src/monitor_task.c`
问题描述：
- `Modbus_InternalWriteRegister` 允许在内部路径写寄存器（绕过外部写保护与回调语义），目前被 `monitor_task` 用于心跳（`DEVICE_STATUS`）更新；
- 舵机回调 `Modbus_OnServoTargetChanged` 在解锁后调用 `Servo_Driver_SetAngle`，但对通道有效性和下层 BSP 参数是否做二次校验未作明确保障；
- Heartbeat 使用 `Modbus_InternalWriteRegister` 更新 `DEVICE_STATUS`，此函数不会触发 `on_change` 回调，可能导致外设状态未完全同步或监控遗漏。
风险点：
- 内部写绕过只读保护与回调，若未来寄存器行为依赖 on_change 触发（如外设驱动、外部通知），可能产生状态不一致。
- 舵机驱动链路对 channel/angle 边界依赖上层检查；若上层校验变更或遗漏，BSP 层需有防护以避免非法参数写入造成硬件损伤。
- Heartbeat 借用内部写虽然成本低，但可读性差，未来维护人员可能误以为 DEVICE_STATUS 仅由外部写修改。
改进建议（最小可执行项）：
1. 在 `问题记录` 中明确标注：`Modbus_InternalWriteRegister` 为内部专用（不触发 on_change），并只有监控/自愈逻辑可使用；
2. 为 `Servo_Driver_SetAngle` 和底层 BSP 增加通道有效性检查（或在 driver 层断言），并在 docs 中记录通道范围与期望行为；
3. 在后续重构中考虑把内部写改为接受一个显式标志位（bypass_callbacks），以提高可读性与审计性；
当前状态：**✅ 已审查并文档化**


### P1-008 UART 驱动互斥/阻塞使用风险
 **符合当前分支主题**（系统防御加固 - 驱动安全）
涉及模块：`App/Driver/Src/driver_uart.c`、`Bsp/Src/uart.c`
问题描述：
- `UART_Driver_Send` 在任务上下文中使用 `osMutexWait(s_uart_mutex, timeout)`，并在持有互斥量期间调用 `BSP_UART_Send`（其内部使用 `HAL_UART_Transmit` 阻塞 API）。
- 互斥量超时值使用调用者传入的 `timeout`（通常为 ms 级），若配置不一致或值过小，可能导致发送失败或短时抢占后频繁重试。
风险点：
- 如果未来有其他任务在持有该互斥量时执行会导致发送请求被拒绝或长时间等待；若 `BSP_UART_Send` 阻塞较长（例如等待硬件），则会占用互斥量，影响其它发送路径。
- 若有人误在中断上下文或高优先级回调中调用驱动发送（应禁止），会产生不可预期行为。
最小改进建议：
1. 在 `driver_uart.c` 的注释中强制声明：该 API 只能在任务上下文调用，禁止 ISR 调用；在文档中标注调用约定。
2. 使用明确的互斥超时常量（如 `UART_DRIVER_MUTEX_TIMEOUT_MS`）并在 `system_config.h` 或 `uart.h` 中定义，避免不同调用方传入不一致的超时语义。
3. 长期改进：将发送路径改为非阻塞提交（环形队列 + 发送任务或信号量）以避免持有互斥量期间阻塞业务任务。
验证清单：
- 本地静态检查：确认所有调用点均在任务上下文（无 ISR 调用 driver_send）。
- 功能测试：在多任务并发发送场景下（2~3 个任务同时发送），验证互斥竞争下无死锁且失败率可接受。
- 长时测试：连续发送 1 小时，观察因互斥导致的发送失败/丢帧统计。

### P1-009 UART 接收轮询与延迟窗口
 **符合当前分支主题**（系统防御加固 - 接收链路健壮性）
涉及模块：`App/Driver/Src/driver_uart.c`（`UART_Driver_Receive`）、`App/Task/Src/uart_task.c`
问题描述：
- `UART_Driver_Receive` 在超时等待模式下采用 `while` 循环 + `osDelay(1)` 的轮询模型，`uart_task` 在无帧时也会 `osDelay(1)`。
- 虽然项目对背靠背请求做了优化，但轮询仍会引入最多 1 ms 的额外延迟窗口，且在高频场景下会增加 CPU 空转。
风险点：
- 在高并发/低间隔的 Modbus 主机请求场景下，轮询导致的 1 ms 延迟可能影响实时性并增加响应延迟。
- 轮询也使得任务难以快速响应硬件事件，建议由中断直接通知任务（信号量/任务通知）以实现事件驱动。
最小改进建议：
1. 在 `BSP_UART_FinalizeFrameFromISR` 或 `HAL_UART_RxCpltCallback` 中使用 `xSemaphoreGiveFromISR` 或 `vTaskNotifyGiveFromISR` 通知 `uart_task`，并在 `UART_Driver_Receive` 中使用等待 API（如 `xSemaphoreTake` 或 `ulTaskNotifyTake`）替代 `osDelay` 轮询。
2. 若短期不改动，文档中明确说明 `osDelay(1)` 的设计意图与可接受的最大延迟，并在测试中测量实际延迟分布。
验证清单：
- 延迟测量：使用回环测试或外部主机连续两帧背靠背发送，记录从 IDLE 到任务处理完帧的延迟分布（包括最大/均值/95% 分位）。
- 功能回归：在事件驱动实现后，验证连续高频请求场景下不丢帧，响应时间低于原轮询模型。
