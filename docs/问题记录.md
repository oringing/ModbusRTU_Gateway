### P1-008 UART 驱动互斥/阻塞使用风险
涉及模块：`App/Driver/Src/driver_uart.c`、`Bsp/Src/uart.c`
问题描述：
- `UART_Driver_Send` 在任务上下文中使用 `osMutexWait(s_uart_mutex, timeout)`，并在持有互斥量期间调用 `BSP_UART_Send`（其内部使用 `HAL_UART_Transmit` 阻塞 API）。
- 互斥量超时值使用调用者传入的 `timeout`（通常为 ms 级），若配置不一致或值过小，可能导致发送失败或短时抢占后频繁重试。
风险点：
- 如果未来有其他任务在持有该互斥量时执行会导致发送请求被拒绝或长时间等待；若 `BSP_UART_Send` 阻塞较长（例如等待硬件），则会占用互斥量，影响其它发送路径。
- 若有人误在中断上下文或高优先级回调中调用驱动发送（应禁止），会产生不可预期行为。
最小改进建议：
1. 在 `driver_uart.c` 的注释中强制声明：该 API 只能在任务上下文调用，禁止 ISR 调用；在文档中标注调用约定。
2. 使用明确的互斥超时常量（如 `UART_DRIVER_MUTEX_TIMEOUT_MS`）并在 `system_config.h` 或 `uart.h` 中定义，避免不同调用方传入不一致的超时语义。
3. 长期改进：将发送路径改为非阻塞提交（环形队列 + 发送任务或信号量）以避免持有互斥量期间阻塞业务任务。
验证清单：
- 本地静态检查：确认所有调用点均在任务上下文（无 ISR 调用 driver_send）。
- 功能测试：在多任务并发发送场景下（2~3 个任务同时发送），验证互斥竞争下无死锁且失败率可接受。
- 长时测试：连续发送 1 小时，观察因互斥导致的发送失败/丢帧统计。

### P1-009 UART 接收轮询与延迟窗口
涉及模块：`App/Driver/Src/driver_uart.c`（`UART_Driver_Receive`）、`App/Task/Src/uart_task.c`
问题描述：
- `UART_Driver_Receive` 在超时等待模式下采用 `while` 循环 + `osDelay(1)` 的轮询模型，`uart_task` 在无帧时也会 `osDelay(1)`。
- 虽然项目对背靠背请求做了优化，但轮询仍会引入最多 1 ms 的额外延迟窗口，且在高频场景下会增加 CPU 空转。
风险点：
- 在高并发/低间隔的 Modbus 主机请求场景下，轮询导致的 1 ms 延迟可能影响实时性并增加响应延迟。
- 轮询也使得任务难以快速响应硬件事件，建议由中断直接通知任务（信号量/任务通知）以实现事件驱动。
最小改进建议：
1. 在 `BSP_UART_FinalizeFrameFromISR` 或 `HAL_UART_RxCpltCallback` 中使用 `xSemaphoreGiveFromISR` 或 `vTaskNotifyGiveFromISR` 通知 `uart_task`，并在 `UART_Driver_Receive` 中使用等待 API（如 `xSemaphoreTake` 或 `ulTaskNotifyTake`）替代 `osDelay` 轮询。
2. 若短期不改动，文档中明确说明 `osDelay(1)` 的设计意图与可接受的最大延迟，并在测试中测量实际延迟分布。
验证清单：
- 延迟测量：使用回环测试或外部主机连续两帧背靠背发送，记录从 IDLE 到任务处理完帧的延迟分布（包括最大/均值/95% 分位）。
- 功能回归：在事件驱动实现后，验证连续高频请求场景下不丢帧，响应时间低于原轮询模型。

3. ✅ 循环退出后 LED 常亮指示，停止喂狗等待看门狗复位系统

## 二、P2 级 中期架构与可维护性优化（迭代改进）

### P2-001 modbus.c 单体文件过大，模块耦合偏高
涉及模块：App/Protocol 协议层  
问题描述：协议解析、寄存器映射、异常响应全部堆砌在单个文件，职责不拆分。  
核心风险：可读性差、耦合严重、新增功能易牵一发而动全身，不利于后期扩展主机、多功能码。  
改进建议：  
1. 模块化拆分：modbus_core 协议帧处理、modbus_map 寄存器映射、modbus_excp 异常响应；  
2. 严格分层：驱动层只提供收发硬件接口，协议层不直接操作外设句柄。  
当前状态：部分处理

### P2-003 缺少单元测试、静态检查与自动化流程
涉及模块：工程整体开发流程  
问题描述：无核心算法单元测试、无静态代码检查、无编译以外的质量管控。  
核心风险：功能迭代后回归测试靠人工，容易隐性引入 bug，版本质量不可控。  
改进建议：  
1. 对 CRC 计算、Modbus 帧解析、异常响应做简易单元测试；  
2. 引入静态语法检查工具做代码规范校验；  
3. 建立基础编译+静态检查的简易版本发布流程。  
当前状态：未处理

### P2-005 系统缺少内存、栈、缓冲区资源运行时监控
涉及模块：系统监控任务、全局资源  
问题描述：仅基础栈水位打印，无整体资源占用、缓冲区负载、内存趋势监控。  
核心风险：长期运行无法提前预判栈溢出、缓冲区爆满、内存持续增长等隐患。  
改进建议：  
1. 在监控任务中常驻统计各任务栈剩余、缓冲区使用率；  
2. 资源临近阈值主动打印告警日志；  
3. 留存运行资源快照，便于后期问题复盘。  
当前状态：未处理

### P2-006 系统日志开关职责耦合，无法细粒度控制
涉及模块：App/System/Inc/system_config.h、App/System/Src/error_handler.c、App/System/Src/system_ctrl.c  
问题描述：[SYSTEM_UART_TEXT_LOG_ENABLE](file://d:\STM32\myproject\ModbusRTU_Gateway\App\System\Inc\system_config.h#L18-L18) 宏同时控制错误日志（Error_SendText）和系统监控日志（System_Monitor_Log）两类不同用途的输出，违反单一职责原则。  
核心风险：
1. 生产环境需要保留错误日志（用于远程故障诊断），但希望关闭监控日志（减少UART负载），当前设计无法实现
2. 错误日志和监控日志的开关策略不同（错误日志生产环境建议启用，监控日志生产环境建议关闭）
3. 一个开关控制两类日志，增加维护复杂度和配置出错风险  
改进建议：  
1. 拆分为两个独立宏：`SYSTEM_ERROR_LOG_ENABLE`（错误日志）和 `SYSTEM_MONITOR_LOG_ENABLE`（监控日志）
2. [Error_SendText()](file://d:\STM32\myproject\ModbusRTU_Gateway\App\System\Src\error_handler.c#L13-L22) 改用 `SYSTEM_ERROR_LOG_ENABLE`
3. [System_Monitor_Log()](file://d:\STM32\myproject\ModbusRTU_Gateway\App\System\Src\system_ctrl.c#L185-L195) 改用 `SYSTEM_MONITOR_LOG_ENABLE`
4. 生产环境建议配置：`SYSTEM_ERROR_LOG_ENABLE=1U`, `SYSTEM_MONITOR_LOG_ENABLE=0U`  
当前状态：**📝 已记录（2026-05-17）**
- **影响范围**：
  - error_handler.c：Error_SendText() 函数（错误记录日志、安全模式告警）
  - system_ctrl.c：System_Monitor_Log() 函数（配置校验、任务管理、栈水位、看门狗日志）
  - 可能涉及 uart_task.c 中的栈水位日志输出
- **改造方案**：
  ```c
  // system_config.h
  #define SYSTEM_ERROR_LOG_ENABLE   1U  // 错误日志开关（生产环境建议启用）
  #define SYSTEM_MONITOR_LOG_ENABLE 0U  // 监控日志开关（生产环境建议关闭）
  ```
- **测试验证**：需在独立分支完成后验证两类日志独立控制功能
